# AI 协同开发 SOP 文档

## 1. 适用范围

本 SOP 适用于以下场景：

- 使用 Cursor / Trae / Claude Code / 其他 Vibe Coding 工具进行前后端开发
- 当前项目已进入“重构+迭代”阶段，而不是纯从零开始
- 项目已有用户端、管理端、API 平台基础能力
- 后续目标是升级为“能力平台 + 任务平台 + 配置驱动发布平台”

本 SOP 适用对象：

- 你自己
- 协作开发者
- AI 编码工具
- 产品 / 前端 / 后端 / 测试协同过程

------

## 2. 核心原则

### 2.1 四条铁律

1. **先设计，后编码**
2. **先检查，后提交**
3. **先复用，后新增**
4. **先小步可控，后扩展放量**

### 2.2 五个禁止项

禁止出现以下情况：

- 没有任务输入模板，直接让 AI 写代码
- 没有开发前检查，直接进入实现
- 没有设计确认，直接生成页面
- 没有 AI 自检，直接提交代码
- 没有开发后自检和人工验收，直接合并上线

### 2.3 一条总链路

每个功能必须经过这条链路：

**需求进入 → 任务标准化 → 开发前检查 → 方案设计 → 设计确认 → 生成代码 → AI 强制自检 → 开发后自检 → 人工/自动验收 → 合并发布 → 上线观察**

少任何一步，风险都会明显升高。

------

## 3. 生命周期总览

把整个 AI 协同开发流程分成 6 个阶段：

### Phase A：基础设计系统搭建

一次性建立全项目共同遵守的规则层。

### Phase B：版本/迭代计划准备

把重构方案拆成可执行任务池。

### Phase C：单任务开发循环

每一个页面、组件、模块都按同样的闭环执行。

### Phase D：联调与验收

验证功能是否真正可用，而不只是“代码已生成”。

### Phase E：发布与灰度

控制上线风险。

### Phase F：上线反馈回流

把问题重新转成下一轮任务输入。

------

# Phase A：基础设计系统搭建

这个阶段只需要完整做一轮，后续补充维护即可。
它是所有后续开发任务的**绝对前置条件**。

------

## A1. 建立 Design Tokens

### 目标

统一项目视觉基础，避免 AI 每次生成不同风格页面。

### 前置条件

- 已明确产品风格：SaaS + AI 产品风格，简洁、可信、现代
- 已知产品主要端：用户端、管理端

### 执行 Prompt

使用你已整理的《设计系统 Step 1：设计 Design Tokens》文档。它要求 AI 设计颜色、字体、间距、圆角、阴影，适配 SaaS + AI 风格。

### 需要再添加什么

建议你在执行时补充：

- 当前项目使用的 UI 库
- 是否已有品牌主色
- 是否已有暗色模式需求
- 用户端和管理端是否共用 Token

建议追加说明：

- 输出需包括变量命名建议
- 输出需区分用户端 Token 与管理端 Token 的复用关系
- 输出需给出 CSS / Tailwind / SCSS 映射建议

### 执行结果怎么落地

结果必须沉淀成项目内的真实文件，例如：

- `tokens/colors.ts`
- `tokens/spacing.ts`
- `tokens/typography.ts`
- `docs/design-tokens.md`

### 是否作为下一步前置条件

**是。**
组件库、页面模板、标准样板页面都依赖 Token。

------

## A2. 建立组件库结构

### 目标

统一组件分层，禁止 AI 到处新造组件。

### 前置条件

- A1 已完成
- 已知项目当前使用的技术栈和 UI 组件基础库

### 执行 Prompt

使用《设计系统 Step 2：组件库结构》。它要求 AI 设计基础组件、业务组件、布局组件，并给出命名规范与目录结构。

### 需要再添加什么

执行时补充：

- 现有项目组件目录
- 当前已存在的公共组件
- 是否允许引入第三方组件库

建议再加限制：

- 不允许和现有组件功能重叠
- 新结构必须说明“迁移方式”
- 输出目录时区分：基础组件、业务组件、页面块组件、布局组件

### 执行结果怎么落地

结果必须变成：

- `src/components/base/`
- `src/components/business/`
- `src/components/layout/`
- `docs/component-architecture.md`

同时输出一份“已存在组件 → 新分层归位表”。

### 是否作为下一步前置条件

**是。**
页面模板和编码实现必须依赖它。

------

## A3. 建立页面模板

### 目标

把页面从“每次现想”变成“按模板套用”。

### 前置条件

- A1、A2 已完成
- 已梳理出平台主要页面类型：列表页、详情页、表单页、任务页

### 执行 Prompt

使用《设计系统 Step 3：页面模板》，它要求定义列表页、详情页、表单页、任务页，并说明页面结构、组件组成、数据流。

### 需要再添加什么

补充：

- 当前项目用户端主要页面类型
- 管理端主要页面类型
- 是否支持 schema 驱动表单

建议追加：

- 每个模板给出空态、加载态、错误态
- 给出桌面端优先还是响应式优先
- 指明“能力页”和“任务页”是一级模板

### 执行结果怎么落地

结果必须落成：

- `docs/page-templates.md`
- 页面骨架组件，如 `ListPageShell`、`DetailPageShell`、`TaskPageShell`
- 页面模板示意图或组件结构图

### 是否作为下一步前置条件

**是。**
开发前检查和方案设计都要检查是否有对应模板。你的开发前检查表已经明确“是否有对应页面模板”是必查项。

------

## A4. 建立开发规则文档

### 目标

给 AI 一个硬边界，避免自由发挥。

### 前置条件

- A1、A2、A3 已完成

### 执行 Prompt

使用《设计系统 Step 4：开发规则文档》，它要求定义命名规范、组件使用规范、状态管理规范、API 调用规范、页面结构规范、禁止事项。

### 需要再添加什么

补充：

- 当前项目实际状态管理方案
- 当前 API 请求封装方式
- 路由命名习惯
- 文件命名习惯
- 目录组织规则

建议再追加：

- 明确禁止直接写散落样式
- 明确禁止重复封装功能等价组件
- 明确“旧 API 平台页面向能力/任务平台迁移”的命名规则

### 执行结果怎么落地

结果必须变成项目内正式文档：

- `docs/dev-rules.md`

并在仓库根目录放一份 AI 必读文件，如：

- `AI_RULES.md`

### 是否作为下一步前置条件

**是。**
后续“开发循环 Step 2 确认”和“Step 4 AI 强制自检”都依赖它。

------

## A5. 生成标准样板页面

### 目标

让 AI 后续“照着写”，而不是“从零想”。

### 前置条件

- A1 到 A4 已完成

### 执行 Prompt

使用《设计系统 Step 5：标准样板页面》，它要求生成能力列表页、任务列表页、能力详情页，并且必须使用前面定义的组件体系、符合页面模板。

### 需要再添加什么

建议补充：

- 当前项目已有页面中哪些能作为迁移基础
- 样板页优先做用户端还是管理端
- 输出需包含目录落点和可复用区块说明

### 执行结果怎么落地

结果必须成为真实参考实现：

- `src/pages/ability/AbilityListTemplate`
- `src/pages/job/JobListTemplate`
- `src/pages/ability/AbilityDetailTemplate`

它们不一定直接上线，但必须作为全项目模板母版存在。

### 是否作为下一步前置条件

**建议是。**
至少“列表页”和“任务页”开发前必须有样板页。

------

## A6. 建立任务输入模板

### 目标

把每个开发任务标准化，避免描述模糊。

### 前置条件

- A1 到 A5 已完成

### 执行 Prompt

使用《设计系统 Step 6：任务输入模板》，它要求输出功能名称、所属模块、页面位置、数据结构、交互流程、依赖组件、API 接口、验收标准。

### 需要再添加什么

建议你加两列：

- 是否影响现网
- 是否依赖其他任务先完成

### 执行结果怎么落地

沉淀为标准任务卡模板，例如：

- `docs/task-template.md`
- `tasks/feature-xxx.md`

### 是否作为下一步前置条件

**是，必须。**
以后所有开发任务必须按此模板进入任务池。

------

# Phase B：版本/迭代计划准备

这个阶段是把重构方案变成“可开发排期”。

------

## B1. 建立版本任务池

### 目标

把重构方案拆成一批可以独立执行的任务。

### 前置条件

- 已有重构方案
- Phase A 完整可用

### 准备什么

按任务输入模板，把需求拆成一条条任务卡。

### 执行什么

每个任务卡必须至少包含：

- 功能名
- 所属端
- 页面
- 依赖组件
- API
- 验收标准
- 优先级
- 是否影响现网
- 前置任务

### 结果是什么

形成：

- `Sprint Backlog`
- `Milestone Backlog`

### 是否作为下一步前置条件

**是。**
没有任务卡，不进入单任务开发循环。

------

## B2. 建立任务依赖图

### 目标

避免 AI 同时开发互相依赖的功能导致结构冲突。

### 前置条件

- B1 已完成

### 执行什么

给每个任务标记：

- 先决任务
- 可并行任务
- 阻塞任务

例如：

- 任务中心页依赖任务页模板和任务列表接口
- 能力详情页依赖 AbilityCard 和详情页模板

### 结果是什么

得到一份任务依赖关系表。

### 是否作为下一步前置条件

建议是。
至少高优先级任务必须先梳理依赖。

------

# Phase C：单任务开发循环

这是你以后每天都会反复执行的核心闭环。

------

## C1. 任务进入开发前检查

### 目标

确认任务已经具备开发条件。

### 前置条件

- Phase A 完成
- 任务卡完整

### 执行 Prompt

使用你的《开发前检查表》。它要求检查：

- 是否已有类似组件
- 是否有对应页面模板
- 是否有清晰任务设计
- API 是否定义清楚
- 数据结构是否确定

### 需要再添加什么

建议在 Prompt 前加：

- 当前任务卡全文
- 当前相关组件文件路径
- 当前相关 API 文件路径
- 当前相关页面模板路径

### 执行结果怎么落地

输出不能只停留在聊天文本，必须记录成：

- 任务状态：Ready / Blocked / Need Info
- 阻塞原因
- 补充项

建议写入任务卡头部。

### 是否作为下一步前置条件

**是。**
未通过开发前检查，不进入设计。

------

## C2. 方案设计（不写代码）

### 目标

先出方案，再实现。

### 前置条件

- C1 已通过

### 执行 Prompt

使用《开发循环 Step 1：任务设计》，它要求只输出页面结构、组件拆分、数据结构、API 设计、状态管理，不写代码。

### 需要再添加什么

建议加：

- 本任务所属页面模板
- 本任务可复用组件清单
- 本任务涉及的状态流转
- 本任务约束：不能改动哪些现有模块

### 执行结果怎么落地

产出一份“小型设计稿”，建议写入：

- `tasks/feature-xxx-design.md`

内容至少包括：

- 目标页面结构
- 组件清单
- 事件流
- API 调用点
- 状态设计
- 风险点

### 是否作为下一步前置条件

**是。**

------

## C3. 设计确认

### 目标

确认方案符合项目规则，而不是只看起来“合理”。

### 前置条件

- C2 已完成
- 有开发规则文档

### 执行 Prompt

使用《开发循环 Step 2：确认》，它要求检查刚才设计是否符合项目开发规则文档，并指出问题。

### 需要再添加什么

建议你在执行时增加约束：

- 是否违反组件复用原则
- 是否绕过页面模板
- 是否引入新模型却未声明
- 是否和能力/任务平台目标冲突

### 执行结果怎么落地

结果必须分级：

- Passed
- Passed with changes
- Rejected

如果不是 Passed，就必须回到 C2 修正。

### 是否作为下一步前置条件

**是。**
未通过确认，不允许编码。

------

## C4. 生成代码

### 目标

在清晰约束下生成当前任务代码。

### 前置条件

- C3 Passed

### 执行 Prompt

使用《开发循环 Step 3：生成代码》，它要求：

- 必须使用已有组件库
- 必须符合页面模板
- 不允许新增未经定义的组件
- 严格遵守开发规则文档、组件结构和页面模板，不允许随意新增结构或破坏现有体系

### 需要再添加什么

每次生成代码前，建议在 Prompt 前附加：

- 当前任务卡
- 当前设计稿
- 涉及文件清单
- 禁止修改的文件清单
- 本次输出范围（例如只修改页面和 API hook，不动全局布局）

### 执行结果怎么落地

代码必须直接落到分支或工作区，且配套输出：

- 修改文件列表
- 新增组件列表
- 新增 API 列表
- 是否涉及旧功能影响

### 是否作为下一步前置条件

**是。**
必须先有代码结果，才能自检。

------

## C5. AI 强制自检

### 目标

让 AI 先给自己做一次“代码审查”。

### 前置条件

- C4 已生成代码

### 执行 Prompt

使用《开发循环 Step 4：AI 强制自检》，它要求检查：

- 是否符合开发规则文档
- 是否重复组件
- 是否违反命名规范
- 是否结构合理
- 是否可维护，并输出问题和修改建议

### 需要再添加什么

建议你强制要求 AI 多输出两项：

- 必修复问题
- 可后续优化问题

并要求：

- 对必修复问题直接给出修正补丁
- 修正后再次复查

### 执行结果怎么落地

输出必须沉淀到任务记录里：

- 本轮发现的问题
- 已修复的问题
- 残留问题
- 风险备注

### 是否作为下一步前置条件

**是。**
未完成 AI 强制自检，不进入开发后自检。

------

## C6. 开发后自检

### 目标

站在“整个项目”的角度做提交前总检查。

### 前置条件

- C5 已通过

### 执行 Prompt

使用你的《开发后自检表》，其中明确要求检查：

- 是否符合组件规范
- 是否符合命名规范
- 是否复用已有组件
- 是否有重复逻辑
- 页面结构是否统一
- 是否符合能力平台/任务平台模型
- 是否影响现有功能
- 是否有异常处理

### 需要再添加什么

建议再补三项：

- 是否有 loading/empty/error 态
- 是否兼容现有权限控制
- 是否需要补测试用例

### 执行结果怎么落地

必须输出成一份提交前清单：

- Pass / Fix Needed / Reject

建议把结果附在 PR 描述或任务卡下方。

### 是否作为下一步前置条件

**是。**

------

## C7. 人工或自动检查

### 目标

验证代码不是“看起来没问题”，而是真的能用。

### 前置条件

- C6 已通过
- 代码能本地运行或在测试环境运行

### 执行 Prompt

你已有《开发循环 Step 5：人工或自动检查》，它和 Step 4 类似，仍然会检查规则、重复、结构和可维护性。
但在真实环境中，这一步还需要扩展成“验收检查”。

### 需要再添加什么

建议你建立一个固定的验收清单，至少包括：

- 页面是否能打开
- 主路径是否可走通
- API 是否通
- 任务是否能提交 / 查询 / 查看结果
- 样式是否破坏布局
- 旧功能是否回归异常
- 控制台是否报错
- 是否需要联调后端

### 执行结果怎么落地

输出：

- 通过验收
- 退回修复
- 待联调
- 待产品确认

### 是否作为下一步前置条件

**是。**
未通过验收，不允许合并。

------

# Phase D：联调与验收

------

## D1. 联调准备

### 目标

保证前后端接口和数据结构一致。

### 前置条件

- 单任务开发完成
- 涉及接口已存在或有 mock

### 准备什么

- API 文档
- 请求参数
- 响应结构
- 错误码
- mock 数据或测试环境地址

### 执行什么

让 AI 或人工逐项核对：

- 接口参数是否匹配
- 字段名是否一致
- 状态流是否对得上
- 错误态是否处理

### 结果是什么

- 联调通过
- 接口待修
- 前端待修
- 后端待修

### 是否作为下一步前置条件

是，涉及接口的功能必须联调后再提测或灰度。

------

## D2. 验收签收

### 目标

把“能跑”变成“可交付”。

### 前置条件

- D1 已通过
- 页面和逻辑通过人工/自动检查

### 执行什么

产品或你自己确认：

- 是否达到任务验收标准
- 是否满足当前重构目标
- 是否可以进入本轮发布列表

### 结果是什么

- Accepted
- Accepted with notes
- Rejected

### 是否作为下一步前置条件

是，未验收通过不进入上线流程。

------

# Phase E：发布与灰度

------

## E1. 发布前检查

### 目标

在进入真实环境前做最后一轮风险控制。

### 前置条件

- 已通过联调和验收

### 准备什么

至少确认：

- 改动范围
- 是否影响数据库
- 是否影响现有 API
- 是否可灰度
- 是否可回滚

### 执行什么

给 AI 或人工生成一份发布前摘要：

- 修改模块
- 风险点
- 回滚方式
- 监控项

### 结果是什么

发布说明文档。

### 是否作为下一步前置条件

是。

------

## E2. 灰度上线

### 目标

减少一次性全量发布的风险。

### 前置条件

- E1 完成
- 系统支持功能开关或最小范围发布

### 执行什么

建议按顺序：

1. 测试环境
2. 内部账号
3. 白名单用户
4. 小流量灰度
5. 全量

### 结果是什么

- 灰度成功
- 回滚
- 修复后重发

### 是否作为下一步前置条件

是，成功后才进入观察期。

# 4. 每一步之间的衔接关系

这是最关键的一部分。你要求“每一步结果是不是作为下一步前置条件”，这里我给你一个直接可执行的依赖图：

## 基础层依赖

- A1 → A2 → A3 → A4 → A5 → A6
  其中 A4 和 A6 是后续所有开发循环的硬前置条件。

## 任务层依赖

- 任务卡完成 → C1 开发前检查 → C2 方案设计 → C3 设计确认 → C4 生成代码 → C5 AI 强制自检 → C6 开发后自检 → C7 人工/自动检查 → D1 联调 → D2 验收 → E1 发布前检查 → E2 灰度上线 → F1 上线观察

任何一个环节失败，都不能默认继续，必须回退。

# 5. 给 Vibe Coding 工具的执行规则

以后你让 Vibe Coding 工具开发时，建议固定在每次任务开头附加这段“执行约束”：

你必须严格遵守本项目的：

1. Design Tokens
2. 组件库结构
3. 页面模板
4. 开发规则文档
5. 标准样板页面
6. 当前任务输入模板

本次任务不允许跳过以下步骤：

- 先做任务设计，不直接写代码
- 先通过设计确认
- 再生成代码
- 生成后必须做 AI 强制自检
- 自检后必须做开发后自检
- 未通过检查不得进入下一步

本次输出必须明确：

- 修改文件
- 新增组件
- 数据流
- API 调用点
- 风险点

# 6. 真实环境落地建议

为了让这套 SOP 真正在工程中落地，我建议你再配三个“操作层制度”。

## 6.1 文档归档制度

所有 AI 生成结果不能只停留在聊天窗口，必须落到仓库里：

- `docs/`
- `tasks/`
- `ai-prompts/`
- `ai-reviews/`

## 6.2 分支制度

每个任务单独分支：

- `feature/ability-list-page`
- `feature/job-center-page`

不要多个任务共用一个 AI 会话同时改一个分支。

## 6.3 PR 模板制度

每个 PR 都必须写：

- 任务名称
- 任务卡链接
- 是否通过开发前检查
- 是否通过 AI 强制自检
- 是否通过开发后自检
- 风险点
- 验收截图/录屏

# 7. 最终执行版本：最简可落地清单

执行顺序是：

1. 完成 Phase A 六个基础文档和母版
2. 用任务输入模板建立本轮重构任务池
3. 每个任务先跑开发前检查
4. 再跑任务设计
5. 再跑设计确认
6. 再让 AI 生成代码
7. 再跑 AI 强制自检
8. 再跑开发后自检
9. 再做人工/自动检查
10. 联调、验收、灰度上线
11. 上线后收集问题重新入池

整个项目的 AI 协同开发闭环。

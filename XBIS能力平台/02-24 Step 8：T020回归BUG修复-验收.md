# 🎯 目标

基于回归测试报告，对当前系统执行：

# ❗Bug 修复 + 自检 + 再验收（完整闭环）

------

# 📦 输入

1. 当前任务卡（tasks/T020-plan-purchase-flow.md）
2. 当前代码实现
3. 当前设计方案（Final）（tasks/design-specs/final/T020-plan-purchase-flow-design-final.md）
4. 回归测试报告（含 Bug 列表）（tasks/result/T020-plan-purchase-flow-retest-acceptance.md）

------

# ⚠️ 修复原则（必须遵守）

- 必须优先修复 High / Medium Bug
- 不允许破坏已有正确逻辑
- 不允许引入新 Bug
- 必须符合组件规范 / Design Tokens / 状态规范
- 修复必须最小影响（禁止大规模重写）

------

# 🔥 Step 1：Bug 分类

从回归报告中提取：

## ❌ High（必须修复）

## ⚠️ Medium（必须修复）

## ℹ️ Low（可选优化）

------

# 🔥 Step 2：逐个 Bug 修复（核心）

对每个 Bug 必须输出：

### Bug：{编号}

- 问题描述
- 根因分析
- 修复方案
- 修改文件
- 修复代码（完整可运行）

------

# 📊 Step 3：修复结果汇总

## 修复清单

| Bug编号 | 优先级 | 状态 |
| ------- | ------ | ---- |
|         |        |      |

------

## 修改文件列表

- 文件路径
- 修改类型（fix / refactor）

------

# 🔍 Step 4：修复后自检（必须执行）

------

## 4.1 功能检查

- 主流程是否正常
- 是否符合任务卡验收标准

------

## 4.2 状态检查

必须具备：

- loading
- empty
- error

------

## 4.3 结构检查

- 是否符合组件分层
- 是否复用组件
- 是否无重复逻辑

------

## 4.4 性能检查

- 是否存在重复渲染
- useCallback / memo 是否合理

------

## 输出：

### ❌ 仍存在问题（如有）

### ⚠️ 可优化问题

------

# 🔍 Step 5：回归复测（关键）

重新执行：

- 主流程测试
- Bug 复现路径测试
- API 调用测试
- UI/状态测试
- 旧功能回归

输出：

| 测试项 | 结果 | 说明 |

------

# 🔍 Step 6：最终验收

👉 输出：

- Accepted
- Accepted with notes
- Rejected

------

# ⚠️ 规则

- 不允许跳过 High Bug
- 不允许模糊描述
- 必须工程化表达

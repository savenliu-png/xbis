# 📋 xbis-front 产品功能审计与界面功能梳理

---

## 第一部分：项目总体判断

### 1.1 当前项目更像什么系统

**明确判断：当前项目是一个"API 开放平台"（API Gateway / API Marketplace 模式）**

核心特征：
- 用户通过 API Key 认证调用接口
- 接口以"市场"形式展示和申请
- 计费模式围绕调用次数/套餐
- 管理端审核、监控、配置接口权限
- 整体思路是"提供方提供 API，消费方通过 Key 调用"

### 1.2 当前项目主要面向哪些角色

| 角色            | 入口               | 核心诉求                                                     |
| --------------- | ------------------ | ------------------------------------------------------------ |
| **开发者/用户** | 用户端（`/`）      | 浏览接口、申请权限、创建 API Key、调用接口、查看记录、管理账单 |
| **平台管理员**  | 管理端（`/admin`） | 审核用户、管理 Token、配置接口、监控调用、处理账单/发票      |

### 1.3 当前主价值

**为用户提供 API 接口调用能力，并提供完整的计费、审核、监控闭环**

### 1.4 当前产品表达

**偏 API 平台 + 管理系统的混合体**

- **API 平台特征**：API 市场、API Key 管理、接口调用记录、接口权限申请
- **管理系统特征**：用户审核、发票审核、认证审核、系统配置、驾驶舱监控
- **能力平台特征（萌芽）**：调用支持 sync/async/review-only 三种模式、执行时间线展示（已具备任务流转雏形）

### 1.5 当前成熟度判断

**MVP 已完成，处于"功能完善期"**

- ✅ 用户注册/登录/认证流程完整
- ✅ API Key 全生命周期管理完整（创建/编辑/禁用/删除/锁定）
- ✅ 调用记录查询与详情完整（含 Timeline 时间线）
- ✅ 计费、账单、充值、代金券、发票流程已搭建
- ✅ 管理端审核、监控、配置功能已具备
- ⚠️ 接口市场、文档中心等运营向功能仍需完善
- ⚠️ 缺少"任务中心"视角，当前调用记录偏向"日志"而非"任务管理"
- ⚠️ 能力配置化程度不足，接口发布仍依赖硬编码路由

## 第六部分：当前业务主流程梳理

### 6.1 用户主流程（端到端链路）

```
1. 访问平台 → /register 注册 → 邮箱密码 → 登录成功 → /login → 跳转控制台 /
   
2. 控制台 / → 查看账户状态、指标卡片、快捷操作
   
3. 浏览能力 → /api-market 接口市场 → /api-market/:apiId 详情 → 查看接口信息
   
4. 申请权限 → 提交接口开通申请 → 管理端审核 → 审核通过 → 接口权限开通
   
5. 创建凭证 → /api-keys 创建 API Key → 选择接口权限 → 配置 IP/过期时间 → 创建成功展示完整 Key
   
6. 发起调用 → 使用 API Key + 接口端点发起调用（sync/async/review-only）
   
7. 查看结果 → /invocations 调用记录 → 查看 Job 详情 → 执行时间线 Timeline → 请求/响应体
   
8. 管理计费 → /subscription 切换套餐 → /billing 查看余额/流水 → 充值申请
   
9. 费用抵扣 → /coupons 兑换代金券 → 套餐切换时选择代金券抵扣
   
10. 申请发票 → /invoices 提交发票申请 → 管理端审核 → 开具发票
```

### 6.2 购买与计费流程

```
套餐购买链路：
用户浏览 /subscription 可选套餐 → 点击"查看报价" → 弹出报价弹窗
→ 选择代金券（可选） → 系统计算：原始金额 - 剩余权益抵扣 - 代金券抵扣 = 应付金额
→ 确认交易 → 余额充足则直接扣款 / 余额不足则提示充值
→ 生成订单 → 订阅历史更新

充值链路：
用户点击 /billing"立即充值" → 输入金额 + 备注 → 提交充值申请
→ 生成 RechargeOrder（状态 pending）→ 管理端审核确认到账
→ 余额增加 → 生成 Transaction 流水记录

消费扣费链路：
每次 API 调用 → 计算费用（基于 QuotaRule 单价配置）
→ 扣减余额 → 生成 Transaction 流水
→ 写入 Consumption 消费明细（按接口聚合）
```

### 6.3 调用/提交流程

```
同步调用（sync）：
用户通过 API Key 发起请求 → 系统接收 → 路由到目标接口 → 执行 → 返回结果 → 扣费 → 完成
状态流转：accepted → routing → running → completed（或 failed）

异步调用（async）：
用户发起请求 → 系统接收 → 进入队列 → 分配执行器 → 执行 → 回调（如开启）
状态流转：accepted → queued → running → completed（或 failed）
用户通过 /invocations 查看状态

复核模式（review-only）：
用户发起请求 → 系统接收 → 进入 Dry Run 预览 → 等待人工复核
→ 管理端 /reviews 人工复核 → 通过/拒绝 → 执行或取消
状态流转：accepted → waiting_review → completed（或 failed/rejected）
```

### 6.4 管理端发布/审核/配置流程

```
接口发布流程：
管理员配置接口 → 设置接口权限策略（/admin/api-permissions）
→ 设置配额规则（/admin/quota） → 配置到套餐允许接口（/admin/plans）
→ 用户在接口市场可见

审核流程：
接口开通申请 → /admin/reviews 待处理列表 → 查看详情 → 通过/拒绝 + 备注
人工复核 → /admin/reviews 人工复核 Tab → 查看请求快照/Dry Run → 通过/拒绝
认证审核 → /admin/certification-review → 查看认证材料 → 通过/拒绝
发票审核 → /admin/invoice-reviews → 查看发票申请 → 通过/拒绝 + 开具

配置流程：
系统配置 /admin/settings → 修改基础/安全/接入/复核/风控配置 → 立即生效
套餐管理 /admin/plans → 创建/编辑套餐 → 配置价格/额度/允许接口 → 启用/停用
配额规则 /admin/quota → 新增规则 → 选择类型/目标/配额值/超额策略 → 生效
代金券 /admin/coupon-batches → 创建批次 → 生成券码 → 发放给用户
```

```

  能力（Capability）= 接口（API）+ 描述元数据 + 使用场景 + 输入输出 schema

  能力草稿 → 能力审核 → 能力发布 → 能力生效

```

## 第十部分：重构起点建议

以下是从产品架构和投入产出比角度，建议**优先改造的页面与模块清单**：

---

### 1. 调用记录（Invocations）→ 任务中心

| 项目             | 内容                                                         |
| ---------------- | ------------------------------------------------------------ |
| **为什么优先改** | 这是整个系统中最接近"任务平台"雏形的页面。已有 jobId、timeline 时间线、多状态流转（accepted→queued→running→waiting_review→completed/failed），本质已经是任务模型，但当前定位为"调用记录"偏日志视角，用户理解成本高，且缺少操作能力（取消/重试）和筛选功能 |
| **改造目标**     | ① 改菜单名为"调用记录"或"任务中心"；② 增加搜索筛选（接口/状态/时间范围）；③ 增加分页；④ 增加任务操作（取消运行中任务、重试失败任务）；⑤ 增加顶部统计卡片（任务总数/成功率/平均耗时/总费用）；⑥ 详情中请求/响应体增加结构化展示 |
| **改造后收益**   | 用户从"查看日志"升级为"管理任务"，是能力平台的核心使用场景；为后续引入能力/任务模型奠定数据基础和交互基础；同时解决无筛选/无分页的性能问题 |

---

### 2. 接口市场（ApiMarket）→ 能力广场

| 项目             | 内容                                                         |
| ---------------- | ------------------------------------------------------------ |
| **为什么优先改** | 这是用户发现和了解平台能力的核心入口。当前更像"接口目录"而非"市场"，缺少分类标签、使用量排行、收藏等运营属性。用户进入后难以快速评估哪些能力适合自己，决策成本高 |
| **改造目标**     | ① 改菜单名为"能力广场"；② 增加分类/标签筛选；③ 列表增加使用量/评分展示；④ 详情页增加调用示例代码（cURL/Python/Node.js）、权限申请状态展示、收藏功能；⑤ 增加面包屑导航（从有侧边栏的页面跳转到独立页面需保留导航上下文） |
| **改造后收益**   | 降低用户发现和使用能力的门槛，提升能力曝光和转化率；为"能力"概念的建立提供产品入口支撑；示例代码直接降低开发者的接入成本 |

---

### 3. 审核与复核（Reviews）→ 任务处理中心

| 项目             | 内容                                                         |
| ---------------- | ------------------------------------------------------------ |
| **为什么优先改** | 这是管理端处理"能力申请"和"人工复核"的核心页面，也是管理端最接近任务工作流的地方。当前两类审核（接口开通申请 + 人工复核）性质完全不同却放在同一个 Tab 中；缺少任务分配、批量审批、SLA 超时提醒等管理功能；人工复核详情以 JSON 展示，非技术人员阅读困难 |
| **改造目标**     | ① 改菜单名为"审核中心"或"任务处理"；② 拆分 Tab 为"能力申请"、"人工复核"、"认证审核"三类（或整合 CertificationReview）；③ 增加任务分配/转交功能；④ 增加批量审批；⑤ 增加 SLA 超时提醒；⑥ 请求快照/Dry Run 结果改为结构化展示 |
| **改造后收益**   | 管理端审核效率大幅提升，审核任务不遗漏不延误；为管理端"任务视角"奠定基础；结构化展示降低审核人员理解成本，减少误判 |

---

### 4. Dashboard 配置化改造

| 项目             | 内容                                                         |
| ---------------- | ------------------------------------------------------------ |
| **为什么优先改** | Dashboard 是用户和管理员登录后的第一页面，指标卡片和引导提示完全依赖后端返回或前端写死，后续新增指标或调用模式都需要改前端代码，迭代成本高。同时通知列表无分页，数据量大时有性能风险 |
| **改造目标**     | ① 指标卡片增加前端 fallback 逻辑（后端无数据时展示默认卡片）；② 引导提示改为配置化返回（后端返回标题/描述/图标/链接数组）；③ 通知列表增加分页；④ 确认"在线调试"快捷按钮功能是否存在，修复链接或移除按钮 |
| **改造后收益**   | 后续新增指标或模式无需改前端，降低迭代成本；通知分页解决性能问题；快捷操作按钮可用性提升 |

### 5. 术语统一与命名规范化

| 项目             | 内容                                                         |
| ---------------- | ------------------------------------------------------------ |
| **为什么优先改** | 当前存在多处命名不一致和命名与实际功能不符的问题：① 用户端叫"API Key"，管理端叫"Token"，概念混淆；② "接口接入"菜单实际展示的是调用记录；③ "限额与计费"页面只有配额规则无计费配置；④ "审核与复核"边界不清。这些问题持续影响用户和管理员的理解效率 |
| **改造目标**     | ① 管理端"Token 管理"→"API Key 管理"；② 用户端"接口接入"→"调用记录"；③ 管理端"限额与计费"→"配额规则"；④ 管理端"审核与复核"→"审核中心"；⑤ 全局将"接口"术语逐步替换为"能力"（第一阶段仅改 UI 文案，不改数据结构） |
| **改造后收益**   | 降低用户和管理员的理解成本，术语一致性提升专业感；为后续"能力平台"升级做好术语铺垫；改动成本低（主要是文案替换），收益立竿见影 |

---

### 6. Billing 财务中心体验优化

| 项目             | 内容                                                         |
| ---------------- | ------------------------------------------------------------ |
| **为什么优先改** | 财务是用户最关心的模块之一。当前充值仅支持线下审核模式，用户体验断层；四个 Tab 平铺（余额流水/充值申请/账单订单/消费明细）信息密度高，用户理解成本偏高；消费明细仅按接口维度聚合，缺少时间/Key 维度分析；账单和消费记录无导出功能 |
| **改造目标**     | ① 改菜单名为"财务中心"；② 增加消费分析图表（按时间趋势展示）；③ 增加账单/消费记录导出功能（CSV/PDF）；④ 充值增加在线支付选项（支付宝/微信）或至少增加充值状态实时通知；⑤ 考虑将 4 个 Tab 重组为更清晰的信息架构（如"余额概览"、"充值记录"、"消费分析"、"账单订单"） |
| **改造后收益**   | 提升财务透明度和用户信任度；在线支付缩短充值到账周期，提升资金周转效率；导出功能满足企业用户财务对账需求 |

### 7. 系统配置变更日志

| 项目             | 内容                                                         |
| ---------------- | ------------------------------------------------------------ |
| **为什么优先改** | 系统配置直接影响整个平台的运行行为（安全策略、接入策略、复核策略、风控规则），但当前无变更记录，不知道谁在什么时候改了什么。一旦配置出错，无法追溯和回滚，运营安全风险大 |
| **改造目标**     | ① 新增配置变更日志表（记录操作人、操作时间、配置分类、改前值、改后值）；② 在 Settings 页面增加"变更记录" Tab 或弹窗查看历史；③ 二期增加配置版本管理和回滚能力 |
| **改造后收益**   | 提升运营安全性，配置变更可追溯可审计；降低配置出错后的排查成本；为后续配置回滚功能奠定数据基础 |

### 8. 隐藏页面导航优化

| 项目             | 内容                                                         |
| ---------------- | ------------------------------------------------------------ |
| **为什么优先改** | 认证审核（CertificationReview）和账户设置（Account）都是隐藏页面，无侧边栏菜单项，仅通过其他页面的链接跳转。管理员可能遗漏认证审核任务，用户不易找到账户设置入口 |
| **改造目标**     | ① 认证审核整合到"审核中心"作为一个 Tab，或增加独立菜单项；② 账户设置增加侧边栏菜单项（可放在底部），或改为"账户中心"整合更多信息（密码修改、安全设置、登录日志） |
| **改造后收益**   | 审核任务不遗漏；用户更容易找到账户设置入口，提升自助服务能力 |

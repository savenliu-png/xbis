------

# 🎯 目标

当前任务已经进行过一次修复，但仍未通过验收。

请执行：

# ❗第二轮精准修复（Iteration Fix）

------

# 📦 输入

1. 当前任务卡
2. 当前代码（修复后的版本）
3. 当前设计方案（Final）
4. 最新验收报告（D2）
5. 上一轮修复记录（如有）

------

# ⚠️ 修复原则（必须遵守）

- 禁止重复上一轮已修复问题
- 禁止破坏已正确部分
- 必须只修复“仍然存在的问题”
- 必须修复“新引入的问题”
- 优先修复高优先级问题

------

# 🔥 Step 1：问题对比分析（关键）

请分析：

------

## 1️⃣ 未修复问题（Residual Issues）

👉 上一轮存在，但仍未解决的问题

------

## 2️⃣ 新引入问题（Regression Issues）

👉 本轮修复引入的新问题

------

## 3️⃣ 已正确部分（Do Not Touch）

👉 标记哪些代码不能再修改

------

# 🔥 Step 2：精准修复（核心）

对于每个问题：

必须输出：

- 问题类型（残留 / 新问题）
- 原因分析
- 修复方案
- 修改代码（必须完整）

------

# 📊 Step 3：输出修复结果

------

## 一、问题分类

| 类型 | 问题 | 状态 |
| ---- | ---- | ---- |
|      |      |      |

------

## 二、修改文件列表

- 文件路径
- 修改原因

------

## 三、修复代码（必须完整）

👉 不允许只给片段

------

# 🔍 Step 4：强制自检（再次验证）

------

## 4.1 功能完整性

- 是否满足任务卡要求
- 是否覆盖所有功能点

------

## 4.2 状态完整性

必须检查：

- loading
- empty
- error

------

## 4.3 结构与规范

- 是否符合组件分层
- 是否复用组件
- 是否符合命名规范
- 是否使用 Design Tokens

------

## 4.4 类型与质量

- 是否存在 any
- 是否类型完整
- 是否可维护

------

## 输出：

### ❌ 仍存在问题（如果有）

### ⚠️ 可优化问题

------

# 🔍 Step 5：最终判断

👉 输出：

是否可以重新进入验收：

- YES（可以再验收）
- NO（仍需修复）

------

# ⚠️ 规则

- 不允许重新大规模重写代码
- 不允许忽略残留问题
- 不允许模糊描述
- 必须给出可运行代码
- 必须工程化表达

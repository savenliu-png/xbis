## 第一部分：项目总体判断

### 1.1 当前项目更像什么系统

**明确判断：当前项目是一个"API 开放平台"（API Gateway / API Marketplace 模式）**

核心特征：
- 用户通过 API Key 认证调用接口
- 接口以"市场"形式展示和申请
- 计费模式围绕调用次数/套餐
- 管理端审核、监控、配置接口权限
- 整体思路是"提供方提供 API，消费方通过 Key 调用"

### 1.2 当前项目主要面向哪些角色

| 角色 | 入口 | 核心诉求 |
|------|------|----------|
| **开发者/用户** | 用户端（`/`） | 浏览接口、申请权限、创建 API Key、调用接口、查看记录、管理账单 |
| **平台管理员** | 管理端（`/admin`） | 审核用户、管理 Token、配置接口、监控调用、处理账单/发票 |

### 1.3 当前主价值

**为用户提供 API 接口调用能力，并提供完整的计费、审核、监控闭环**

### 1.4 当前产品表达

**偏 API 平台 + 管理系统的混合体**

- **API 平台特征**：API 市场、API Key 管理、接口调用记录、接口权限申请
- **管理系统特征**：用户审核、发票审核、认证审核、系统配置、驾驶舱监控
- **能力平台特征（萌芽）**：调用支持 sync/async/review-only 三种模式、执行时间线展示（已具备任务流转雏形）

### 1.5 当前成熟度判断

**MVP 已完成，处于"功能完善期"**

- ✅ 用户注册/登录/认证流程完整
- ✅ API Key 全生命周期管理完整（创建/编辑/禁用/删除/锁定）
- ✅ 调用记录查询与详情完整（含 Timeline 时间线）
- ✅ 计费、账单、充值、代金券、发票流程已搭建
- ✅ 管理端审核、监控、配置功能已具备
- ⚠️ 接口市场、文档中心等运营向功能仍需完善
- ⚠️ 缺少"任务中心"视角，当前调用记录偏向"日志"而非"任务管理"
- ⚠️ 能力配置化程度不足，接口发布仍依赖硬编码路由

## 第六部分：当前业务主流程梳理

### 6.1 用户主流程
注册 → 登录 → 控制台 → 浏览接口市场 → 申请接口权限 → 管理端审核通过 → 创建 API Key → 发起调用 → 查看调用记录 → 管理账单/充值

### 6.2 购买与计费流程
浏览套餐 → 查看报价（自动计算升级/降级差额） → 选择代金券抵扣 → 确认交易 → 余额扣款 → 生成订单

### 6.3 调用/提交流程
- **同步调用**：accepted → routing → running → completed/failed
- **异步调用**：accepted → queued → running → completed/failed → 回调
- **复核模式**：accepted → waiting_review → 管理端审核 → completed/failed/rejected

### 6.4 管理端审核/配置流程
接口权限配置 → 配额规则设置 → 套餐管理 → 代金券批次发放 → 审核接口申请/人工复核/认证/发票 → 系统配置调整

### 6.5 最完整链路
1. 注册 → 登录 → 创建 API Key ⭐⭐⭐⭐⭐
2. API Key → 调用 → 查看记录 → Timeline 详情 ⭐⭐⭐⭐⭐
3. 套餐浏览 → 报价计算 → 代金券抵扣 → 交易确认 ⭐⭐⭐⭐⭐
4. 接口申请 → 管理端审核 → 通过/拒绝 ⭐⭐⭐⭐

### 6.6 断裂链路
- 在线调试功能未实现
- 充值仅支持线下审核，无在线支付
- 调用记录无取消/重试操作
- 文档中心可能为硬编码
- 统计数据/账单无导出（Statistics 已有导出功能，账单缺少）

- Dashboard 和 Notifications 页面都有通知列表（设计意图：摘要 vs 完整版）
- Reviews 和 CertificationReview 职责不同（接口申请/复核 vs 身份认证）

### 7.4 用户理解成本问题（6 项）
- API Key "接口范围"与"可调用接口"概念重叠
- 套餐 allowedApiIds 与权限策略双重校验
- 三种调用模式缺少引导文档
- 充值"需后台确认"提示不够明确
- Quota 规则表单复杂
- 人工复核 JSON 展示难理解

### 7.5 API 思维过重问题（5 项）
- 整体以"接口"为能力单元 → 升级为"能力"
- 调用记录偏"日志"视角 → 增加任务视图
- 缺少"能力发布"概念 → 增加发布工作流
- 接口市场缺少运营属性 → 增加评分/排行
- 调用与结果分离 → 增加结构化结果展示

### 7.6 不利于扩展问题（7 项）
- Dashboard 指标卡片无 fallback
- 引导提示写死3种模式
- 文档中心可能硬编码
- 系统配置无关联提示
- 调用记录/用户管理/Token 管理无分页

### 7.7 管理端配置能力不足（7 项）
- 系统配置无变更日志/回滚
- 配额规则无生效时间/模板
- 权限策略无冲突检测
- 套餐无排序配置
- 无全局通知发送能力

## 第十部分：重构起点建议

### 1. 调用记录（Invocations）→ 任务中心（P0）

| 项目 | 内容 |
|------|------|
| 为什么优先改 | 最接近"任务平台"雏形的页面，已有 jobId/timeline/多状态流转，但定位为"日志"缺操作能力 |
| 改造目标 | 改名 + 搜索筛选 + 分页 + 取消/重试操作 + 统计卡片 |
| 改造后收益 | 用户从"查看日志"升级为"管理任务"，是能力平台核心场景 |

### 2. 接口市场（ApiMarket）→ 能力广场（P0）

| 项目 | 内容 |
|------|------|
| 为什么优先改 | 用户发现能力的核心入口，当前缺少分类/使用量/示例代码，决策成本高 |
| 改造目标 | 改名 + 分类标签 + 使用量展示 + 示例代码 + 权限申请状态 |
| 改造后收益 | 降低用户发现和使用能力门槛，提升转化率 |

### 3. 审核与复核（Reviews）→ 任务处理中心（P0）

| 项目 | 内容 |
|------|------|
| 为什么优先改 | 管理端最接近任务工作流的页面，缺少分配/批量/SLA |
| 改造目标 | 改名 + 任务分配 + 批量审批 + SLA 超时提醒 + 结构化展示 |
| 改造后收益 | 管理端审核效率大幅提升，任务不遗漏不延误 |

### 4. Dashboard 配置化改造（P1）

| 项目 | 内容 |
|------|------|
| 为什么优先改 | 指标卡片和引导提示完全依赖后端/写死，迭代成本高 |
| 改造目标 | 指标 fallback + 引导配置化 + 通知分页 + 修复在线调试链接 |
| 改造后收益 | 新增指标/模式无需改前端，降低迭代成本 |

### 5. 术语统一与命名规范（P1）

| 项目 | 内容 |
|------|------|
| 为什么优先改 | 多处命名不一致/不符，持续影响理解效率 |
| 改造目标 | 统一 API Key 命名、接口→能力、菜单重命名 |
| 改造后收益 | 改动成本低（文案替换），收益立竿见影 |

### 6. Billing 财务中心体验优化（P1）

| 项目 | 内容 |
|------|------|
| 为什么优先改 | 充值仅线下审核、无导出、信息密度高 |
| 改造目标 | 在线支付 + 消费图表 + 导出功能 + 信息架构重组 |
| 改造后收益 | 提升财务透明度和资金周转效率 |

### 7. 系统配置变更日志（P1）

| 项目 | 内容 |
|------|------|
| 为什么优先改 | 配置直接影响平台运行，无变更记录无法追溯 |
| 改造目标 | 新增变更日志表 + Settings 页面查看历史 |
| 改造后收益 | 提升运营安全性，配置可追溯可审计 |

### 8. 隐藏页面导航优化（P2）

| 项目 | 内容 |
|------|------|
| 为什么优先改 | 认证审核和账户设置作为隐藏页面不易发现 |
| 改造目标 | 认证审核整合到审核中心或增加菜单项 + 账户设置增加菜单项 |
| 改造后收益 | 审核任务不遗漏，用户易找到设置入口 |

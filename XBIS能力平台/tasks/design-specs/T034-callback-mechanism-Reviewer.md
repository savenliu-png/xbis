## 问题列表

### 问题1: 回调安全机制设计不足
**严重程度: 高**

回调 URL 可能被恶意利用（如 SSRF 攻击），但设计方案仅提到"签名验证；URL 白名单"，未详细说明实现方案。

### 问题2: 回调重试的退避策略未明确
**严重程度: 中**

提到"指数退避；最大 3 次"，但未给出具体的退避时间间隔。

### 问题3: 回调超时时间未设置
**严重程度: 中**

回调请求如果长时间未响应，会占用连接资源，但未设置超时时间。

### 问题4: 回调记录缺少清理策略
**严重程度: 低**

回调记录会持续增长，但未说明数据保留期限和清理策略。

---

## 修改建议

### 建议1: 详细设计回调安全机制（必须修改）
```typescript
// 签名验证
interface CallbackSignature {
  timestamp: string;
  nonce: string;
  signature: string; // HMAC-SHA256(secret, timestamp + nonce + body)
}

// URL 白名单校验
const allowedCallbackDomains = [
  '*.example.com',
  'partner.com'
];
```

### 建议2: 明确退避策略（必须修改）
```typescript
const retryIntervals = [30, 60, 120]; // 秒
// 第1次重试：30秒后
// 第2次重试：60秒后
// 第3次重试：120秒后
```

### 建议3: 设置回调超时（必须修改）
```typescript
const CALLBACK_TIMEOUT = 30000; // 30 秒
// 超过 30 秒未响应视为失败，触发重试
```

### 建议4: 增加数据清理策略（建议修改）
- 回调记录保留 30 天
- 成功回调记录保留 7 天
- 定期清理任务（每天凌晨执行）

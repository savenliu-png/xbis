## 问题列表

### 问题1: 支付流程缺少安全校验机制
**严重程度: 高**

设计方案中未体现支付前的安全校验（如二次确认、防重复提交）。支付是敏感操作，需要更严格的安全措施。

### 问题2: 支付结果回调处理未设计
**严重程度: 中**

用户支付完成后，需要从支付网关跳转回应用。设计方案未说明如何处理支付回调和结果查询。

### 问题3: 优惠券实时计算缺少防抖
**严重程度: 低**

用户输入优惠券码时，每次输入都会触发验证请求，可能导致频繁请求。

### 问题4: 套餐对比功能未详细设计
**严重程度: 中**

PlanComparison 组件提到"套餐对比"，但未说明对比的具体维度（价格/配额/功能）和展示方式。

---

## 修改建议

### 建议1: 增加支付安全校验（必须修改）
- 支付前显示二次确认 Modal
- 创建订单时生成唯一幂等键（idempotency key）
- 订单创建后锁定表单，防止重复提交

### 建议2: 设计支付回调处理流程（必须修改）
```
[支付网关回调]
  ├── 成功回调 ──► 查询订单状态 ──► 展示成功页面
  ├── 失败回调 ──► 展示失败页面 ──► 提供重试入口
  └── 用户手动返回 ──► 轮询订单状态 ──► 展示结果
```

### 建议3: 优惠券输入增加防抖（建议修改）
```typescript
const debouncedValidate = useDebounce(validateCoupon, 500);
```

### 建议4: 明确套餐对比设计（建议修改）
PlanComparison 采用表格对比形式：
- 行：功能特性
- 列：各套餐
- 单元格：✓ / ✗ / 具体数值

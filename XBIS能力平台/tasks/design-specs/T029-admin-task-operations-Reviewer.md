## 问题列表

### 问题1: 审核/退款操作缺少操作日志记录
**严重程度: 中**

审核和退款是敏感操作，但设计方案未说明如何记录操作日志（谁、何时、做了什么操作）。

### 问题2: 批量操作的错误处理不完善
**严重程度: 中**

批量操作中部分任务可能成功、部分失败，设计方案未说明如何展示部分失败的结果。

### 问题3: SLA 统计的计算公式未明确
**严重程度: 低**

SLA 达成率的计算公式未说明（如：成功任务数/总任务数，还是基于响应时间）。

### 问题4: 退款金额缺少校验规则
**严重程度: 中**

退款金额不能超过订单金额，但设计方案未明确校验规则。

## 修改建议

### 建议1: 增加操作日志记录（必须修改）
审核/退款操作后记录日志：
```typescript
interface OperationLog {
  operatorId: string;
  operatorName: string;
  action: 'review' | 'refund' | 'retry';
  targetId: string;
  details: Record<string, unknown>;
  createdAt: string;
}
```

### 建议2: 完善批量操作错误处理（必须修改）
批量操作结果展示：
- 成功数量
- 失败数量
- 失败原因列表（任务ID + 失败原因）
- 支持重试失败项

### 建议3: 明确 SLA 计算公式（建议修改）
```typescript
// SLA 达成率 = 按时完成的任务数 / 总任务数
// 平均响应时间 = 所有任务响应时间的平均值
// 待处理任务数 = 状态为 pending + running 的任务数
// 已超时任务数 = 超过 SLA 时间仍未完成的任务数
```

### 建议4: 增加退款金额校验（必须修改）
```typescript
// 退款金额校验规则
// 1. 退款金额 > 0
// 2. 退款金额 ≤ 订单剩余可退金额
// 3. 退款金额 ≤ 订单总金额
```

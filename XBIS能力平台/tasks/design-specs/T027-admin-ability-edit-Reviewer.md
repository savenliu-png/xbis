## 问题列表

### 问题1: SchemaEditor 和 FormBuilder 复杂度过高
**严重程度: 高**

两个组件都是高复杂度组件，需要支持 JSON Schema 完整语法和拖拽式表单配置。在 1.5 天内完成开发风险极高。

### 问题2: 6 Tab 大表单的性能风险
**严重程度: 中**

所有 Tab 同时挂载（即使非活跃），6 个 Tab 的表单数据量大，可能导致内存占用高和渲染性能问题。

### 问题3: 自动保存与手动保存的冲突未明确
**严重程度: 中**

设计方案同时存在「保存草稿」按钮和「自动保存（Debounce 3s）」，但未说明两者的优先级和冲突处理。

### 问题4: 缺少版本历史/回滚功能
**严重程度: 中**

能力发布后有版本概念，但设计方案未说明如何查看历史版本或回滚到旧版本。

## 修改建议

### 建议1: SchemaEditor 采用渐进式实现（必须修改）
分阶段实现：
- **第一阶段**：支持基础类型（string/number/boolean）
- **第二阶段**：支持对象和数组
- **第三阶段**：支持嵌套和高级属性

或考虑使用开源库 **@rjsf/core** 作为基础进行二次封装。

### 建议2: Tab 内容懒加载（必须修改）
非活跃 Tab 不渲染内容，切换时才挂载：
```typescript
{activeTab === 'schema' && <SchemaTab />}
```

### 建议3: 明确保存策略（必须修改）
- 自动保存：仅保存草稿状态，不提示用户
- 手动保存：保存并提示成功
- 用户点击保存时，如果自动保存正在进行，等待自动保存完成后再执行手动保存

### 建议4: 增加版本历史查看（建议修改）
在 PublishTab 中增加版本历史列表，显示：
- 版本号
- 发布时间
- 发布人
- 变更日志
- 回滚按钮（回滚到该版本）

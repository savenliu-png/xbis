# XBIS 能力平台缺失功能清单

## 一、总体判断

当前已有：

- 用户端能力列表 T011
- 用户端能力详情 T012
- 能力测试面板 T013
- 能力接入流程 T014
- 管理端能力列表 T026
- 管理端能力编辑 T027

但目前这些任务更多覆盖了“能力展示、编辑、测试、接入”的页面结构，缺少从 **XClaw Agent OS 接口注册为能力、发布能力、用户直接使用能力、任务执行、扣费、结果展示** 的完整闭环。

---

# 1. XClaw 接口接入为能力的功能缺失

## 1.1 缺失功能

管理端缺少“把 XClaw Agent OS 接口登记为能力”的完整流程。

## 1.2 应补功能

### 管理端新增：能力接口绑定配置

在 T027 管理端能力编辑页的“执行配置 Tab”中补充：

- 接口来源选择
  - XClaw Agent OS
  - XBIS 内部接口
  - 第三方 API
- 接口地址配置
  - `http://localhost:8000/xxx`
- Method 配置
  - GET / POST / PUT / DELETE
- Header 配置
- Auth 配置
- 请求参数映射
- 响应字段映射
- 超时时间
- 重试次数
- 是否异步执行
- 是否需要回调

## 1.3 建议新增字段

```typescript
export interface AbilityExecutionConfig {
  abilityId: string;
  executorType: 'xclaw' | 'internal' | 'third_party';
  endpointUrl: string;
  method: 'GET' | 'POST' | 'PUT' | 'DELETE';
  headers?: Record<string, string>;
  authType?: 'none' | 'api_key' | 'bearer' | 'basic';
  requestMapping?: Record<string, string>;
  responseMapping?: Record<string, string>;
  executionMode: 'sync' | 'async';
  timeout: number;
  retryCount: number;
}
```

# 2. 能力发布前测试功能缺失

## 2.1 缺失功能

管理端配置完能力后，缺少“发布前真实测试”。

## 2.2 应补功能

在 T027 管理端能力编辑页新增：

- 测试接口连通性
- 测试请求参数
- 查看 XClaw 原始返回
- 查看 XBIS 映射后结果
- 测试失败原因展示
- 保存测试样例
- 测试通过后才允许发布

## 2.3 发布前校验规则

能力发布前必须校验：

- 基础信息完整
- requestSchema 存在
- responseSchema 存在
- uiSchema 存在
- executionConfig 存在
- pricingConfig 存在
- 接口测试通过
- 计费规则有效

# 4. 用户执行能力时自动创建任务的功能缺失

## 4.1 缺失功能

用户点击“开始执行”后，应自动创建 Job，但当前能力详情、测试、接入之间还没有形成统一执行链路。

## 4.2 应补流程

```text
用户填写参数
→ 点击开始执行
→ 校验余额/套餐
→ 创建 Job
→ 调用 XClaw 接口
→ 返回 jobId
→ 前端轮询任务状态
→ 展示结果
```

## 4.3 建议新增 API

```http
POST /api/v1/abilities/:abilityId/run
```

请求：

```json
{
  "input": {},
  "files": [],
  "confirmBilling": true
}
```

响应：

```json
{
  "jobId": "job_001",
  "status": "queued",
  "estimatedCost": 0.5
}
```

# 5. 能力执行计费功能缺失

## 5.1 缺失功能

T027 有计费配置字段，T014 有接入/套餐流程，但缺少“普通用户直接使用能力时扣费”的闭环。

## 5.2 应补功能

能力执行前：

- 查询余额
- 查询免费额度
- 查询套餐额度
- 计算本次费用
- 展示价格确认

执行时：

- 冻结费用或扣除额度
- 创建计费流水
- 绑定 jobId

执行后：

- 成功确认扣费
- 失败退款或释放冻结额度
- 生成消费记录

## 5.3 建议新增 API

```http
POST /api/v1/billing/estimate
POST /api/v1/billing/charge
POST /api/v1/billing/refund
```

------

# 6. 能力结果展示功能缺失

## 6.1 缺失功能

能力执行后，用户需要直接看到结果，而不是只看到任务状态。

## 6.2 应补功能

ResultPanel 支持：

- JSON 结果展示
- Markdown 结果展示
- 表格结果展示
- 文件下载
- 图片预览
- 复制结果
- 重新执行
- 查看原始输出

## 6.3 依赖 responseSchema

根据能力的 `responseSchema` 自动选择结果展示方式。

------

# 7. XClaw 返回结果映射功能缺失

## 7.1 缺失功能

XClaw 接口返回结构可能和 XBIS 能力结果结构不一致，需要映射层。

## 7.2 应补功能

管理端配置：

```json
{
  "xclaw.result.text": "output.reportText",
  "xclaw.result.summary": "output.summary",
  "xclaw.file.url": "output.downloadUrl"
}
```

## 7.3 建议新增能力字段

```typescript
export interface AbilityResponseMapping {
  sourcePath: string;
  targetPath: string;
  transform?: 'string' | 'number' | 'json' | 'file_url';
}
```

# 8. 能力订阅与直接使用关系不清晰

## 8.1 当前问题

T014 更偏开发者接入流程：套餐、密钥、文档。

但普通 ToC 用户不应该必须先接入 API Key 才能使用能力。

## 8.2 应拆成两种模式

### 普通用户模式

```text
能力详情页 → 直接使用 → 按次扣费
```

### 开发者模式

```text
能力详情页 → 接入能力 → 获取 API Key → 程序调用
```

## 8.3 页面按钮建议

能力详情页应有两个主要按钮：

- `立即使用`
- `开发者接入`

# 14. 开发者调用能力 API 缺少统一网关

## 14.1 缺失功能

如果开发者用 API Key 调用能力，应该调用 XBIS 统一能力接口，而不是直接调用 XClaw。

## 14.2 推荐接口

```http
POST /api/v1/abilities/:abilityId/invoke
GET /api/v1/jobs/:jobId
GET /api/v1/jobs/:jobId/result
```

调用链：

```text
开发者 API Key
→ XBIS 鉴权
→ 计费
→ 创建 Job
→ 调用 XClaw
→ 返回 jobId
```

# 16. 最终应形成的完整闭环

## 管理端

```text
新建能力
→ 填基础信息
→ 配置 XClaw 接口
→ 配置 requestSchema / responseSchema
→ 配置 UI 表单
→ 配置计费规则
→ 测试接口
→ 发布能力
→ 监控能力
→ 下架 / 回滚
```

## 用户端普通用户

```text
能力中心
→ 能力详情
→ 填写参数 / 上传文件
→ 查看价格
→ 点击执行
→ 扣费
→ 创建任务
→ 等待执行
→ 查看结果
```

## 开发者用户

```text
能力详情
→ 开发者接入
→ 选择套餐
→ 获取 API Key
→ 调用 XBIS 能力 API
→ 查询任务结果
→ 按量计费
```

# 18. 最关键结论

当前 T011-T014、T026-T027 已经覆盖了能力平台的页面基础，但还缺少：

```text
XClaw接口 → 能力配置 → 发布 → 用户执行 → 任务创建 → 扣费 → 执行 → 结果展示
```

这一条真正能产生商业价值的主链路。

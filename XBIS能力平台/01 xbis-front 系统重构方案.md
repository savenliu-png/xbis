## 二、系统重构总体策略

### 2.1 核心策略：**分层升级 + 双轨并行**

```
┌─────────────────────────────────────────────────────────────┐
│  目标架构：能力平台 + 任务平台 + 配置发布平台                  │
├─────────────────────────────────────────────────────────────┤
│  用户端层：首页 → 能力中心 → 任务提交 → 我的任务 → 结果查看   │
│  管理端层：概览 → 能力管理 → 执行器管理 → 任务运营 → 发布系统  │
├─────────────────────────────────────────────────────────────┤
│  中间层（新增）：                                             │
│    - 能力层（Ability）：包装接口为产品化能力                   │
│    - 任务层（Job）：统一调用为任务模型                         │
│    - 配置层（Config）：schema 驱动 UI 和发布                  │
├─────────────────────────────────────────────────────────────┤
│  现有层（保留）：                                             │
│    - 接口层（API）：原有调用链路不变                           │
│    - 计费层（Billing）：原有计费逻辑不变                       │
│    - 权限层（Auth）：原有认证授权不变                          │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 关键决策

| 决策项               | 决策                  | 理由                                            |
| -------------------- | --------------------- | ----------------------------------------------- |
| 是否新增"任务层"     | ✅ 是                  | 现有 `Job` 类型已具备任务雏形，只需增强页面视角 |
| 是否新增"能力层"     | ✅ 是                  | 在 `ApiMarketItem` 基础上扩展产品化字段，不替换 |
| 原有"接口层"如何保留 | ✅ 保留为底层          | 所有能力最终仍通过接口调用，接口层不变          |
| 是否影响现有功能上线 | ❌ 不影响              | 采用"新增页面 + 路由切换"方式，旧页面保留       |
| 如何做到逐步上线     | ✅ 功能开关 + 路由灰度 | 通过系统配置控制新页面是否可见                  |

### 2.3 数据流设计

```
用户端：
  用户 → 能力中心 → 选择能力 → 查看详情（uiSchema渲染表单）
  → 提交任务 → 创建 Job → 返回任务ID
  → 我的任务 → 查询 Job 状态 → 查看结果

管理端：
  管理员 → 能力管理 → 创建/编辑能力（填写基础信息/inputSchema/outputSchema/uiSchema）
  → 绑定执行器 → 配置价格 → 预览 → 灰度发布 → 全量上线
  → 任务运营 → 查看所有 Job → 重试/取消/退款
```

### 3.3 用户主流程改造

**原流程（API调用导向）：**
```
注册/登录 → 控制台 → 创建API Key → 查看接口市场 → 申请接口权限
→ 等待审核 → 使用API Key调用接口 → 查看调用记录 → 管理账单
```

**新流程（任务结果导向）：**
```
注册/登录 → 首页 → 浏览能力中心 → 进入能力详情 → 查看示例
→ 填写输入（uiSchema表单） → 提交任务 → 跳转我的任务
→ 查看任务进度（Timeline） → 查看结果 → 下载/分享
```

**双轨并存（保留开发者路径）：**
```
开发者路径：首页 → 开发者中心 → 创建API Key → 查看文档 → 调试 → 接入
```

### 4.3 能力发布机制设计（重点）

**目标：新增能力不用改代码，不用重启系统**

#### 4.3.1 Ability Schema 设计

```typescript
// 在现有 ApiMarketItem 基础上扩展
interface Ability {
  // === 基础信息（现有字段复用）===
  id: string;                    // 复用 apiId
  name: string;                  // 复用 apiName
  displayName: string;           // 复用 displayName
  description: string;           // 复用 description
  category: string;              // 复用 category
  icon?: string;                 // 新增
  tags?: string[];               // 复用 tags
  status: 'draft' | 'testing' | 'internal' | 'whitelist' | 'published' | 'deprecated';
  
  // === 输入输出定义（新增）===
  inputSchema: JSONSchema;       // 新增：输入参数schema
  outputSchema: JSONSchema;      // 新增：输出结果schema
  fileLimits?: {                 // 新增：文件限制
    maxSize: number;
    allowedTypes: string[];
  };
  
  // === UI 配置（新增）===
  uiSchema: {                    // 新增：表单渲染配置
    fields: Array<{
      name: string;
      type: 'text' | 'textarea' | 'select' | 'file' | 'number' | 'date';
      label: string;
      placeholder?: string;
      required?: boolean;
      options?: Array<{label: string; value: any}>;
      validation?: any;
    }>;
    layout: 'single' | 'two-column';
  };
  
  // === 执行配置（新增）===
  executorId: string;            // 新增：绑定执行器
  fallbackExecutorId?: string;   // 新增：fallback执行器
  timeoutMs: number;             // 复用 executionTimeoutMs
  retryCount: number;            // 新增
  queueType: 'normal' | 'priority';
  maxConcurrency: number;        // 复用 concurrencyLimit
  
  // === 计费配置（复用现有）===
  pricingType: 'free' | 'per_call' | 'package' | 'overage';
  unitPrice?: number;            // 复用 unitPrice
  freeQuota?: number;            // 复用 freeQuota
  
  // === 发布记录（新增）===
  currentVersion: string;
  versions: AbilityVersion[];
  releaseNotes?: string;
  
  createdAt: string;
  updatedAt: string;
}
```

#### 4.3.2 执行器绑定

```typescript
interface Executor {
  executorId: string;
  name: string;
  type: 'internal' | 'external' | 'agent_os';
  source: 'self' | 'third_party';
  
  // 连接配置
  endpoint: string;              // 复用 backendTargetAddress
  authType: 'api_key' | 'oauth' | 'none';
  authConfig?: Record<string, any>;
  
  // 能力绑定
  supportedAbilities: string[];  // 该执行器支持的能力列表
  
  // 健康检查
  healthCheckUrl?: string;
  healthStatus: 'healthy' | 'unhealthy' | 'unknown';
  lastHealthCheck?: string;
  
  // 限流
  rateLimit?: string;
  
  createdAt: string;
  updatedAt: string;
}
```

#### 4.3.3 发布流程

```
后台操作：
  1. 创建能力（填写基础信息）
  2. 配置输入输出 schema
  3. 配置 UI schema（表单渲染）
  4. 绑定执行器
  5. 配置价格和权限
  6. 预览用户端样式
  7. 保存为草稿
  
  8. 灰度发布（选择白名单用户）
  9. 监控灰度效果
  10. 全量上线 / 回滚
```

**关键技术点：**
- 用户端能力详情页根据 `ability.uiSchema` 动态渲染表单（使用 `react-jsonschema-form` 或自研渲染器）
- 任务提交时根据 `ability.inputSchema` 校验输入
- 任务结果根据 `ability.outputSchema` 格式化展示
- 新增能力只需插入数据库，用户端自动展示

| 表名                  | 改造内容                                                     | 影响                   |
| --------------------- | ------------------------------------------------------------ | ---------------------- |
| **api_market_item**   | 重命名为 ability，增加 ui_schema/input_schema/output_schema/executor_id 字段 | 中：需同步修改所有查询 |
| **invocation_record** | 重命名为 job，增加 ability_id 字段（替代 api_id）            | 中：需同步修改统计查询 |

### 5.3 数据迁移策略

```
第一阶段（上线前）：
  1. 创建新表（ability, executor, ability_version 等）
  2. 将 api_market_item 数据复制到 ability（保留原表）
  3. 将 invocation_record 数据复制到 job（保留原表）
  4. 双写：新代码同时写入新旧表

第二阶段（上线后）：
  5. 验证新表数据一致性
  6. 切换读操作到新表
  7. 停止写入旧表
  8. 保留旧表一段时间（备份）
```

### 6.2 旧接口兼容策略

```
旧接口（/portal-api/v1）：
  - 保留所有现有端点
  - 内部转发到新实现
  - 标记为 deprecated，逐步引导迁移

新接口（/api/v1）：
  - 面向任务模型
  - 面向能力模型
  - 面向配置驱动

切换机制：
  - 通过系统配置控制新旧路由
  - 默认新用户走新接口
  - 老用户可选切换
```

### 6.3 Callback 机制

```
任务完成回调：
  1. 用户提交任务时可选填 callbackUrl
  2. 任务完成后系统发送 POST 请求到 callbackUrl
  3. 回调内容：{ jobId, status, result, completedAt }
  4. 支持重试（最多3次，间隔5分钟）
  5. 管理端可查看回调记录
```

## 九、风险与注意事项

### 9.1 高风险项

| 风险                    | 影响                 | 缓解措施                                         |
| ----------------------- | -------------------- | ------------------------------------------------ |
| **数据迁移失败**        | 导致现有调用记录丢失 | 双写策略：新旧表同时写入，验证一致性后再切换     |
| **uiSchema 渲染器复杂** | 影响能力详情页加载   | 先支持基础字段类型（text/select/file），逐步扩展 |
| **旧接口兼容问题**      | 影响现有用户调用     | 保留所有旧端点，内部转发，标记 deprecated        |
| **导航重构影响**        | 用户找不到原有功能   | 保留旧路由一段时间，增加重定向，提供切换引导     |

### 9.2 灰度上线策略

```
阶段1（内部测试）：
  - 新页面部署到测试环境
  - 内部账号访问新页面
  - 验证功能完整性

阶段2（白名单灰度）：
  - 生产环境部署，默认不可见
  - 通过系统配置开启白名单
  - 选择部分用户试用新页面

阶段3（全量切换）：
  - 监控白名单用户反馈
  - 修复问题后全量开启
  - 保留旧页面入口一段时间（如1个月）

阶段4（下线旧页面）：
  - 确认新页面稳定后
  - 下线旧页面路由
  - 清理旧代码
```

### 9.3 不中断服务保证

| 措施       | 说明                                                        |
| ---------- | ----------------------------------------------------------- |
| **数据库** | 新增表不影响现有表，双写保证数据一致性                      |
| **API**    | 新增 /api/v1 端点，旧端点保留                               |
| **前端**   | 新页面独立路由，旧页面保留，通过功能开关控制可见性          |
| **部署**   | 前后端独立部署，先部署后端（新增API），再部署前端（新页面） |
| **回滚**   | 保留旧版本镜像，出现问题可快速回滚                          |

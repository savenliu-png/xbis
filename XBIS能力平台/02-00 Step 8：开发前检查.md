# ❗判断这个任务是否“允许进入开发阶段”

------

# 🎯 任务目标

对当前任务执行 **Phase C1：开发前检查（Pre-Dev Gate）**

输出必须包含：

👉 是否可以开发（Ready / Blocked / Need Info）
👉 阻塞原因（如果有）
👉 补充项（必须做什么才能开始）
👉 风险提示

------

# 📦 输入内容（必须使用）

1. 当前任务卡（sprint-backlog.md）
2. 相关依赖任务（task-dependencies.md）
3. 当前项目架构（能力平台 + 任务系统）

------

# 🔍 检查维度（必须逐项检查）

------

## 1. 组件与复用检查（避免重复造轮子）

请判断：

- 是否已有可复用组件（base/business/blocks）
- 是否已有类似页面或功能
- 是否违反组件分层（base → business → blocks → layout）

输出：

- 已存在组件
- 是否必须新建组件
- 是否存在重复开发风险

------

## 2. 页面模板检查（UI结构是否统一）

请判断：

- 是否匹配页面模板（ListPageShell / DetailPageShell / FormPageShell）
- 是否存在布局不一致风险

输出：

- 使用的页面模板
- 是否缺失模板
- 是否需要新增模板

------

## 3. 任务设计完整性检查（AI能否正确开发）

请判断：

- 功能描述是否清晰
- 是否存在歧义
- 是否缺少交互说明
- 是否缺少边界条件（空态/错误态）

输出：

- 是否设计完整
- 缺失项列表

------

## 4. API 契约检查（是否可联调）

请判断：

- 是否已有 API（如 `/api/v1/xxx`）
- 请求参数是否明确
- 返回结构是否明确
- 是否有 mock 数据

输出：

- API 是否 Ready
- 缺失字段
- 是否需要 Mock

------

## 5. 数据结构检查（是否会返工）

请判断：

- 数据模型是否已定义（ability / job / executor）
- 字段是否满足页面需求
- 是否存在字段缺失/不一致

输出：

- 数据结构是否完整
- 潜在冲突点

------

## 6. 依赖任务检查（是否被阻塞）

请判断：

- 前置任务是否完成（如 T001/T008 等）
- 是否存在阻塞任务
- 是否可并行开发

输出：

- 前置任务状态
- 是否阻塞
- 可并行任务建议

------

## 7. 风险检查（基于项目风险库）

结合已有风险库，判断：

- 是否命中高风险项（如 uiSchema / 异步任务 / 数据迁移）
- 是否需要降级方案

输出：

- 风险等级（高/中/低）
- 应对建议

------
